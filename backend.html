变量 & 类型 — Java vs Dart

Dart 的类型系统和 Java 非常接近，都是强类型。主要区别在于 Dart 多了 const（编译时常量）和 dynamic（≈ Object）。

J Java
JAVA
final String name = "hello";  // 不可重新赋值
int count = 0;               // 可重新赋值
var old = "legacy";          // Java 10+ 类型推断

static final String API =   // 编译时常量
  "https://api.com";

int age = 25;
Object data = "anything";   // 任意类型

// 基本类型
int, double, boolean, String
List<String>, Map<String,Object>
⇄ 差异说明

Java final = Dart final（运行时常量）

Java static final ≈ Dart const（编译时常量）

Java var = Dart var（类型推断后锁定）

Java Object ≈ Dart dynamic（可随时改类型）

Dart 没有 int/Integer 区分，统一 int

Dart bool 不是 boolean

Dart 字符串用单引号 'hello'

D Dart
DART
final String name = 'hello';  // 运行时常量
int count = 0;               // 可重新赋值
var old = 'legacy';          // 类型推断，锁定

const String API =           // 编译时常量
  'https://api.com';

int age = 25;
dynamic data = 'anything';   // 任意类型

// 基本类型
int, double, bool, String
List<String>, Map<String,dynamic>

函数 — Lambda vs 箭头函数

J Java
JAVA
// 方法

Dart 函数是一等公民，不需要 Function 接口

Dart => 箭头只能写一行

多行用 {} 函数体（和 Java 一样）

Dart 有 async/await（Java 没有）

Dart 函数需声明返回类型（和 Java 一样）

Dart 支持顶层函数（不用放在类里）

空安全 — Optional 的进化版

Java 用 Optional + @Nullable 处理空值，Dart 直接在类型系统层面解决 — 更简洁，更安全。

J Java
JAVA
// Optional 包装
Optional<String> name =
  Optional.ofNullable(user.getName());

String val = name.orElse("default");
int len = name.map(String::length)
  .orElse(0);

// @Nullable 注解提示
public void setName(
  @Nullable String name) { }

// 经典 NPE 防御
if (user != null
    && user.getName() != null) {
  // ...
}
⇄ 差异说明

Dart String? ≈ Optional<String>

Dart ?? ≈ orElse()

Dart ?. ≈ map() 链式调用

Dart ! ≈ get()（强制取值）

Dart ??= 空值赋值（Java 没有）

Dart late ≈ 延迟初始化字段

Dart 默认不可 null，编译器强制检查

D Dart
DART
// 类型后加 ? 表示可空
String? name;        // 可以为 null
String title = '';   // 不可为 null

String val = name ?? 'default';
int? len = name?.length;

// 非空断言（确定不为 null 时）
String n = name!;

// 空值赋值
name ??= 'unknown';

// 延迟初始化
late String token;
// 用之前必须赋值，否则运行时报错

类 & 构造函数

Dart 的类系统对 Java 开发者来说非常亲切。但多了命名构造函数、初始化列表、factory 等语法糖，写起来更简洁。

J Java
JAVA
public class User {
  private final String name;
  private final int age;

  // 构造函数
  public User(String name, int age) {
    this.name = name;
    this.age = age;
  }

  // 重载构造
  public User(String name) {
    this(name, 0);
  }

  // 静态工厂方法
  public static User fromJson(Map json) {
    return new User(
      (String)json.get("name"),
      (int)json.get("age"));
  }

  // getter
  public String getName() {
    return name;
  }
}
⇄ 差异说明

Java 构造重载 → Dart 命名构造函数

Java this.name = name → Dart this.name 语法糖

Java 静态工厂 → Dart User.fromJson()

Java getter → Dart get 关键字

Dart 有初始化列表 :

Dart factory = 工厂模式内置

Dart 没有 public/private 关键字，用 _ 前缀表示私有

D Dart
DART
class User {
  final String name;
  final int age;

  // 语法糖：this.name 自动赋值
  User(this.name, this.age);

  // 命名构造 + 初始化列表
  User.fromJson(Map json)
    : name = json['name'],
      age = json['age'];

  // 命名参数构造
  User.create({
    required this.name,
    this.age = 0,
  });

  // getter（不用写 getName）
  String get displayName =>
    '$name ($age)';
}

🏭 factory 构造 = 单例模式内置

Java 写单例需要 private 构造 + static getInstance()。Dart 用 factory 一行搞定。

JAVAJava 单例
public class HttpManager {
  private static final HttpManager
    INSTANCE = new HttpManager();

  private HttpManager() { }

  public static HttpManager getInstance() {
    return INSTANCE;
  }
}
DARTDart 单例
class HttpManager {
  static final HttpManager _instance =
    HttpManager._internal();

  factory HttpManager() => _instance;

  HttpManager._internal() {
    // 初始化 Dio...
  }
}

参数系统 — 告别 Builder 模式

Java 没有命名参数，复杂对象要用 Builder 模式。Dart 原生支持命名参数，这就是为什么 Flutter Widget 写起来像 Builder 但不用写 Builder 类。

J Java
JAVA
// 只有位置参数
public void greet(String name, int age) {}
greet("Tom", 25);

// 复杂对象 → Builder 模式
User user = User.builder()
  .name("Tom")
  .age(25)
  .email("tom@test.com")
  .build();

// 方法重载模拟可选参数
public void log(String msg) {
  log(msg, 0);
}
public void log(String msg, int level) {}
⇄ 差异说明

Dart {} = 命名参数（内置 Builder）

Dart [] = 可选位置参数

required 标记必传

不用写 Builder 类，不用方法重载

Widget 属性全是命名参数

这就是 Container(padding: ..., color: ...) 的由来

D Dart
DART
// 命名参数 — {} 包裹
void createUser({
  required String name,
  int age = 0,
  String? email,
}) { }

createUser(
  name: 'Tom',
  age: 25,
  email: 'tom@test.com',
);

// 可选位置参数 — [] 包裹
void log(String msg, [int level = 0]) {}
log('hello');
log('hello', 2);

集合操作 — Stream API vs Dart 链式

Java 8 的 Stream API 你肯定很熟。Dart 的集合操作和它非常像，但不需要 .stream() 开头。

J Java
JAVA
List<Integer> list = List.of(1,2,3,4,5);

// map
var doubled = list.stream()
  .map(x -> x * 2)
  .toList();

// filter
var filtered = list.stream()
  .filter(x -> x > 2)
  .toList();

// find
var first = list.stream()
  .filter(x -> x > 3)
  .findFirst().orElse(0);

// Map
Map<String,String> map = Map.of(
  "name", "Tom");
map.get("name");
map.containsKey("name");
⇄ 差异说明

Dart 不需要 .stream() 开头

Java filter → Dart where

Java findFirst → Dart firstWhere

都需要 .toList() 收集结果

Dart Map 用 map['key'] 访问（像数组下标）

删除用 removeWhere（类似 removeIf）

Dart 还有 ...spread 运算符

D Dart
DART
var list = [1, 2, 3, 4, 5];

// map — 不需要 .stream()
var doubled = list
  .map((e) => e * 2)
  .toList();

// where ≈ filter
var filtered = list
  .where((e) => e > 2)
  .toList();

// firstWhere ≈ findFirst
var first = list
  .firstWhere((e) => e > 3,
    orElse: () => 0);

// Map
var map = {'name': 'Tom'};
map['name'];
map.containsKey('name');

异步模型 — CompletableFuture vs Future

Java 的异步用 CompletableFuture + thenApply 链式调用，写起来比较啰嗦。Dart 有 async/await，和写同步代码一样直观。

J Java
JAVA
// CompletableFuture
CompletableFuture<User> future =
  CompletableFuture.supplyAsync(() ->
    restTemplate.getForObject(url, User.class)
  );

// 链式处理
future.thenApply(user -> user.getName())
  .thenAccept(name -> log.info(name));

// 并发
CompletableFuture.allOf(futureA, futureB)
  .join();

// Reactor Flux（响应式流）
Flux<String> stream = webClient
  .get().retrieve()
  .bodyToFlux(String.class);
stream.subscribe(System.out::println);
⇄ 差异说明

Future ≈ CompletableFuture

async/await 让异步像同步一样写

Future.wait ≈ CompletableFuture.allOf

Stream ≈ Reactor Flux / RxJava Observable

.listen() ≈ .subscribe()

async* + yield 生成 Stream

AI 对话 SSE 就是典型 Stream 场景

D Dart
DART
// async/await — 写起来像同步
Future<User> fetchUser() async {
  final res = await dio.get(url);
  return User.fromJson(res.data);
}

// 链式（也支持，但不推荐）
fetchUser().then((user) =>
  print(user.name));

// 并发
final results = await Future.wait([
  fetchA(), fetchB()
]);

// Stream ≈ Flux
Stream<String> sseStream() async* {
  yield 'chunk1';
  yield 'chunk2';
}
stream.listen((data) => print(data));

IoC & 依赖注入 — Spring IoC vs GetX

这是后端同学最容易理解的部分。GetX 的依赖注入和 Spring IoC 思路完全一致：注册 Bean → 获取 Bean。

S Spring Boot
SPRING
// 注册 Bean
@Service
public class UserService { }

@Configuration
public class AppConfig {
  @Bean
  public RestTemplate restTemplate() {
    return new RestTemplate();
  }
}

// 获取 Bean
@Autowired
private UserService userService;

// 懒加载
@Lazy
@Autowired
private HeavyService heavyService;
⇄ 差异说明

@Service → Get.put()

@Bean → Get.put()

@Autowired → Get.find()

@Lazy → Get.lazyPut()

Spring 自动扫描 → GetX 手动注册

Spring 单例默认 → GetX 也是单例

思路完全一致，只是 API 不同

D Flutter + GetX
DART
// 注册（≈ @Service / @Bean）
Get.put(UserController());

路由 — @RequestMapping vs GetPage

Spring 的路由是注解映射，Flutter 的路由是什么？
Flutter 用路由表映射 URL → Widget 页面，和 Spring 的 @RequestMapping 映射 URL → Controller 方法是同一个思路。实际项目推荐 GetX 的命名路由。
S Spring Boot
SPRING
@RestController
public class PageController {

  @GetMapping("/")
  public String home() {
    return "home";
  }

  @GetMapping("/detail/{id}")
  public String detail(
    @PathVariable Long id) {
    return "detail";
  }
}

// 重定向
return "redirect:/home";
⇄ 差异说明

Spring: URL → Controller 方法

Flutter: URL → Widget 页面

@GetMapping → GetPage(name:)

@PathVariable → Get.arguments

redirect: → Get.offNamed()

forward: → Get.toNamed()

D Flutter + GetX
DART
// 路由表（≈ @RequestMapping）
GetMaterialApp(
  getPages: [
    GetPage(
      name: '/',
      page: () => HomePage()),
    GetPage(
      name: '/detail',
      page: () => DetailPage()),
  ],
);

// 跳转（≈ forward）
Get.toNamed('/detail',
  arguments: {'id': 123});

// 替换（≈ redirect）
Get.offNamed('/home');

网络请求 — RestTemplate vs Dio

后端用 RestTemplate / OkHttp / WebClient，Flutter 用什么？
Flutter 用 Dio，功能 ≈ OkHttp + RestTemplate。支持拦截器、超时、FormData、文件上传下载、SSE 流式响应。封装方式和你在 Spring 中封装 RestTemplate 几乎一样。
S Spring Boot
SPRING
// RestTemplate 配置
@Bean
public RestTemplate restTemplate() {
  var rt = new RestTemplate();
  rt.setInterceptors(List.of(
    (req, body, exec) -> {
      req.getHeaders()
        .set("token", token);
      return exec.execute(req, body);
    }
  ));
  return rt;
}

// 调用
User user = restTemplate
  .getForObject(url, User.class);

// 错误处理
try {
  restTemplate.getForObject(url);
} catch (HttpClientErrorException e) {
  if (e.getStatusCode() == 401)
    logout();
}
⇄ 差异说明

Dio ≈ OkHttp + RestTemplate

拦截器思路完全一致

Spring @Bean 配置 → Dio 单例封装

Spring 拦截器 → Dio InterceptorsWrapper

HttpClientErrorException → DioException

超时用 Duration 对象

支持 SSE 流式响应（AI 对话）

D Dio (Flutter)
DART
// Dio 配置（≈ @Bean RestTemplate）
final _dio = Dio(BaseOptions(
  baseUrl: 'https://api.xxx.com',
  connectTimeout:
    Duration(seconds: 10),
));

// 拦截器（≈ Spring Interceptor）
_dio.interceptors.add(
  InterceptorsWrapper(
    onRequest: (options, handler) {
      options.headers['token'] = token;
      handler.next(options);
    },
    onError: (e, handler) {
      if (e.response?.statusCode == 401)
        logout();
      handler.next(e);
    },
  ),
);

// 调用
final res = await _dio.get('/users');

状态管理 — @Service 的前端版

后端没有"响应式 UI"的概念，但你可以这样理解：GetxController ≈ @Service（管理业务逻辑），.obs 变量 ≈ 自动通知前端刷新的字段。

S Spring Boot 类比
SPRING类比理解
// GetxController ≈ @Service
@Service
public class UserService {
  // 状态字段
  private boolean loading = false;
  private List<User> userList;
  private String errorMsg = "";

  // ≈ onInit 生命周期
  @PostConstruct
  public void init() {
    fetchUserList();
  }

  public void fetchUserList() {
    loading = true;
    try {
      userList = restTemplate
        .getForObject(url);
    } catch (Exception e) {
      errorMsg = e.getMessage();
    } finally {
      loading = false;
    }
  }
}
// 但 Spring 不会自动通知 UI 刷新
// GetX 的 .obs 会自动通知！
⇄ 差异说明

@Service → extends GetxController

@PostConstruct → onInit()

普通字段 → .obs 响应式变量

手动返回数据 → Obx() 自动刷新 UI

@Autowired → Get.find()

核心区别：GetX 会自动通知 UI 更新

D Flutter + GetX
DART
class UserController extends GetxController {
  // .obs = 响应式（改了自动刷新 UI）
  final isLoading = false.obs;
  final userList = <Map>[].obs;
  final errorMsg = ''.obs;

  // ≈ @PostConstruct
  @override
  void onInit() {
    super.onInit();
    fetchUserList();
  }

  Future<void> fetchUserList() async {
    isLoading.value = true;
    try {
      final res = await dio.get('/users');
      userList.value = res.data;
    } on DioException catch (e) {
      errorMsg.value = e.message ?? '失败';
    } finally {
      isLoading.value = false;
    }
  }
}

// UI 中 — Obx 自动监听变化
Obx(() => Text('${ctrl.userList.length}'))

UI 组件 — 万物皆 Widget

后端同学可以这样理解：Flutter 的 Widget 就是一棵 Java 对象树。每个 UI 元素都是一个类的实例，通过构造函数的命名参数配置属性。

后端没写过前端 UI，怎么理解 Widget？
想象你在 Java 中构建一棵树：new Column(children: [new Text("Hello"), new Button("Click")])。每个节点是一个对象，属性通过构造函数传入。Flutter 就是这样构建 UI 的 — 没有 HTML，没有 CSS，全是 Dart 对象。
